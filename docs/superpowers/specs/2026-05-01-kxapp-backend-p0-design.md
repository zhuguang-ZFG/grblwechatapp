# KXApp P0 Backend Design

**Purpose:** Define the first backend service that supports the existing WeChat mini program and turns the current mock-driven frontend into a real end-to-end P0 system.

**Scope:** P0 only. The goal is to support registration/login, project creation, preview requests, generation requests, and job tracking with a backend we control. This spec intentionally stops short of real machine protocol integration and full image/vector/GCode processing.

## 1. Product Goal

Build a single backend service inside this repository that the mini program can call directly. The service should expose the API contract already drafted in the plan docs, persist core entities, and simulate async preview/generation/job workflows in a way that preserves the final architecture shape.

The backend must let one user:

- log in with WeChat code exchange semantics
- complete registration/profile flow
- view and bind devices
- create and update projects
- request a preview
- request generation
- submit a job
- inspect job progress and retry/cancel

## 2. P0 Design Principles

1. Use a **single deployable Fastify service** first.
2. Keep the API surface aligned with the existing frontend and plan docs.
3. Use **SQLite** for local persistence so the stack runs with minimal setup.
4. Use **local filesystem storage** for generated artifacts in P0.
5. Model preview, generation, and job execution as **async resources**, even if workers are mocked.
6. Do not build the real device gateway in this phase; simulate gateway behavior through backend workers and state transitions.

## 3. Architecture

### 3.1 Runtime shape

P0 backend is one Node.js process with:

- Fastify HTTP server
- SQLite database access layer
- local storage helper for artifact files
- in-process worker scheduler for async state changes

This gives us a simple local development model while preserving the separation between synchronous API handlers and asynchronous workflow execution.

### 3.2 Module boundaries

The backend should be split into modules by domain:

- `auth`
- `devices`
- `projects`
- `previews`
- `generations`
- `jobs`
- `shared/db`
- `shared/storage`
- `shared/session`
- `workers`

Each module owns:

- route registration
- request/response schemas
- repository functions
- domain-level service logic

## 4. Technology Choices

### 4.1 Framework

- Node.js
- Fastify

Reason:

- small and fast
- good schema-based route ergonomics
- easy to keep scoped for P0

### 4.2 Database

- SQLite in local file mode for P0

Reason:

- no external dependency to start
- easy local iteration
- enough for P0 entity and state tracking

The schema should be written so that moving to PostgreSQL later is practical.

### 4.3 Storage

- local `backend/storage/` directory

Reason:

- simulate object storage without infrastructure dependency
- good enough for preview images, SVG placeholders, and GCode placeholder files

### 4.4 Auth/session

- bearer token session for the mini program
- token can be opaque random session token in P0

Reason:

- easier than full JWT lifecycle for first version
- matches frontend expectation of `Authorization: Bearer <token>`

## 6. Core Data Model

P0 backend should implement these persisted entities:

- `users`
- `sessions`
- `devices`
- `projects`
- `previews`
- `generations`
- `jobs`
- `job_events`

### 6.1 Simplifications for P0

- one user role only
- one selected device per project
- device records can be pre-seeded plus bindable
- preview/generation artifacts can be placeholders generated by the service
- uploaded image handling can remain stubbed until the upload API is truly needed

## 7. API Design

The backend should implement these endpoints first:

### Auth

- `POST /api/v1/auth/wechat-login`
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/complete-profile`
- `GET /api/v1/auth/me`

### Devices

- `GET /api/v1/devices`
- `POST /api/v1/devices/bind`

### Projects

- `POST /api/v1/projects`
- `GET /api/v1/projects`
- `GET /api/v1/projects/:id`
- `PUT /api/v1/projects/:id`

### Previews

- `POST /api/v1/projects/:id/preview`
- `GET /api/v1/previews/:id`

### Generations

- `POST /api/v1/projects/:id/generate`
- `GET /api/v1/generations/:id`

### Jobs

- `POST /api/v1/jobs`
- `GET /api/v1/jobs`
- `GET /api/v1/jobs/:id`
- `POST /api/v1/jobs/:id/cancel`
- `POST /api/v1/jobs/:id/retry`

## 8. Workflow Design

### 8.1 Auth flow

`POST /auth/wechat-login`

- accept WeChat code plus lightweight profile payload
- in P0, treat code exchange as successful if code exists
- if user exists and profile complete: return `token`, `user`, `nextAction=enter_app`
- if user exists but incomplete: return `token`, `user`, `nextAction=complete_profile`
- if user does not exist: create temporary login context and return `nextAction=register`

For P0, temp registration context can be represented in memory or as a short-lived DB row if needed. The frontend already supports a `pendingRegistration` local flow, so backend can stay lean here.

### 8.2 Registration flow

`POST /auth/register`

- create user if not present
- mark `profileStatus=completed`
- create session token
- return token and normalized user object

### 8.3 Preview flow

`POST /projects/:id/preview`

- create preview row with `status=processing`
- enqueue in-process worker
- worker writes preview artifact placeholder file
- worker updates preview to `ready`

P0 preview output can be placeholder-but-structured:

- generated text summary file
- optional SVG placeholder
- preview image placeholder path

The important thing is not realism yet, but persistence and async lifecycle.

### 8.4 Generation flow

`POST /projects/:id/generate`

- create generation row with `status=processing`
- enqueue worker
- worker writes placeholder GCode file and path artifact
- worker updates generation to `ready`

### 8.5 Job flow

`POST /jobs`

- validate project, generation, device relation
- create job row with `status=queued`
- append initial `job_event`
- enqueue worker

Job worker state progression in P0:

1. `queued`
2. `dispatching`
3. `running`
4. `completed`

Failure simulation should also be supported through controlled conditions so the frontend retry flow can be tested.

## 9. Worker Model

P0 uses **in-process fake async workers** rather than an external queue.

Implementation shape:

- route creates DB row
- route schedules `setTimeout` or lightweight task runner
- worker updates DB rows later

This is acceptable for P0 because:

- it keeps setup simple
- it exercises real polling flows from the mini program
- it preserves the future boundary for queue extraction

Worker responsibilities:

- preview artifact generation
- generation artifact generation
- job state transitions

## 10. Device Model

P0 backend should not pretend to implement real hardware communication.

Instead:

- seed a few devices in DB
- support bind by `bindingCode`
- support `online`, `offline`, `busy`, `error`
- when jobs run, temporarily reflect execution-oriented state

This lets the frontend exercise real device selection and status display without depending on gateway completion.

## 11. Error Handling

Every module should return structured JSON errors:

```json
{
  "message": "Human readable summary",
  "code": "machine_readable_code"
}
```

P0 should explicitly cover:

- missing auth token
- invalid or expired session
- project not found
- preview not found
- generation not found
- job not found
- device not found
- invalid retry target
- invalid cancel target

## 12. Testing Strategy

P0 backend tests should cover:

- auth route happy path
- register + complete profile flows
- authenticated route protection
- project CRUD basics
- preview creation and eventual ready state
- generation creation and eventual ready state
- job creation and eventual state transitions
- retry and cancel rules

Test layers:

- focused route/service tests
- one lightweight end-to-end API flow for the P0 closed loop

## 13. Non-Goals

P0 backend explicitly does **not** include:

- real WeChat server-side code exchange
- real object storage
- real Bluetooth or serial machine control
- real gateway service
- full image preprocessing
- true SVG/path conversion from reverse-engineered Android logic
- true GCode generation fidelity

Those all remain future work, but the interfaces should leave room for them.

## 14. Delivery Sequence

1. Scaffold backend app and configuration
2. Add DB bootstrap and schema creation
3. Implement auth routes and session middleware
4. Implement devices routes
5. Implement projects routes
6. Implement previews routes plus fake worker
7. Implement generations routes plus fake worker
8. Implement jobs routes plus fake worker/state machine
9. Point mini program to backend in real mode
10. Verify one end-to-end P0 flow

## 15. Acceptance Criteria

This spec is satisfied when:

- backend boots locally with one command
- frontend can authenticate against it
- projects persist in SQLite
- preview requests return async ids and later resolve to `ready`
- generation requests return async ids and later resolve to `ready`
- jobs can be created, queried, retried, and canceled
- frontend can switch from mock adapter to real adapter and complete the P0 loop

## 16. Future Evolution

After P0:

- replace SQLite with PostgreSQL if needed
- replace local storage with object storage
- replace in-process workers with queue workers
- introduce gateway service as separate runtime
- swap placeholder preview/generation logic with true processing pipeline
